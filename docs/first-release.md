# First Public Release Guide

This guide walks through the full sequence for Suture's first public release:
from a clean local state to a TestPyPI install, with a decision gate before
touching real PyPI.

---

## A. Before GitHub push

Run the full preflight and verify nothing unexpected is staged:

```bash
uv run python scripts/preflight.py
git status
```

The following should **not** appear in `git status` output. If they do, they
are not being ignored correctly:

| Path | Why it must not be committed |
|---|---|
| `dist/` | Built artefacts — regenerated by `uv build` |
| `.tmp/` | Local install-test venvs |
| `.venv/` | Project virtual environment |
| `__pycache__/` | Python bytecode cache |

If any of these appear, check `.gitignore` before continuing.

---

## B. Create the GitHub repository

Create a new repository manually at https://github.com/new.

**Suggested name:** `suture`

**Fallbacks if taken:**
- `suture-cli`
- `suture-py`

Use the description and topics from `docs/repo-description.md`:

> Python project setup doctor for broken imports, env files, pyproject config,
> and pytest path issues.

**Topics to add under Settings → Topics:**

```
python  cli  developer-tools  pyproject  pytest  env  python-packaging  diagnostics
```

Set visibility to **Public**.

Do not initialise with a README, `.gitignore`, or licence — the repo already
has all of these.

---

## C. First commit and push

```bash
git init
git add .
git commit -m "Initial Suture release"
git branch -M main
git remote add origin <your-github-repo-url>
git push -u origin main
```

Replace `<your-github-repo-url>` with the actual URL from GitHub (e.g.
`https://github.com/yourusername/suture.git`).

Verify the push includes:

- `src/`
- `tests/`
- `scripts/`
- `examples/`
- `docs/`
- `.github/`
- `pyproject.toml`
- `uv.lock`
- `README.md`, `LICENSE`, `CHANGELOG.md`
- `.env.example`
- `examples/broken-projects/missing-env-gitignore/.env` (fake fixture, safe)

---

## D. Verify GitHub Actions

After pushing, go to the **Actions** tab on GitHub and wait for the CI
workflow to complete on all three Python versions (3.10, 3.11, 3.12).

**Do not upload to TestPyPI until CI passes.**

If CI fails, fix the issue locally and push again before continuing.

---

## E. TestPyPI upload

Run preflight one final time to confirm everything is clean:

```bash
uv run python scripts/preflight.py
```

Then upload to TestPyPI:

```bash
uv run twine upload --repository testpypi dist/*
```

You will be prompted for credentials. Use a **TestPyPI API token**:

- Create one at https://test.pypi.org/manage/account/token/
- When prompted for username, enter `__token__`
- When prompted for password, paste the full token (starts with `pypi-`)

**Never commit tokens.** Never paste tokens into issue reports, chat, or logs.

---

## F. Install from TestPyPI

After the upload completes, verify the install in a clean environment.

**macOS / Linux:**

```bash
python -m venv .tmp/testpypi-install
.tmp/testpypi-install/bin/python -m pip install --upgrade pip
.tmp/testpypi-install/bin/python -m pip install \
    --index-url https://test.pypi.org/simple/ \
    --extra-index-url https://pypi.org/simple \
    suture
.tmp/testpypi-install/bin/suture doctor --json
```

**Windows (PowerShell):**

```powershell
py -m venv .tmp\testpypi-install
.tmp\testpypi-install\Scripts\python -m pip install --upgrade pip
.tmp\testpypi-install\Scripts\python -m pip install `
    --index-url https://test.pypi.org/simple/ `
    --extra-index-url https://pypi.org/simple `
    suture
.tmp\testpypi-install\Scripts\suture doctor --json
```

Confirm:

- `suture doctor --json` exits 0 and prints parseable JSON
- `suture --help` shows all subcommands
- `suture env`, `suture imports`, `suture pyproject` each run without error

---

## G. External validation before PyPI

Before publishing to real PyPI, run Suture against at least three external
Python projects:

- one small beginner-style project (e.g. a simple script or tutorial repo)
- one `src/` layout package (e.g. a library on GitHub)
- one web/API project such as FastAPI, Flask, or Django

For each project, check:

- Are the confirmed issues genuinely broken, or false positives?
- Are the possible issues fair warnings or misleading?
- Is the wording of reasons and suggestions clear to someone who didn't write
  Suture?

Record any false positives or confusing wording before continuing to PyPI.
This is the last chance to fix output quality before the package is public.

---

## H. PyPI publish decision

Do not publish to real PyPI until all of the following are true:

- [ ] TestPyPI install works cleanly (step F above)
- [ ] GitHub Actions pass on all three Python versions
- [ ] README install section uses "Once published" wording (no false claims)
- [ ] Package name `suture` confirmed available on PyPI (see `docs/package-name-check.md`)
- [ ] At least three external repos scanned with no alarming false positives
- [ ] CHANGELOG updated with the release date (replace `Unreleased`)

When ready:

```bash
uv run twine upload dist/*
```

Use a **PyPI API token** (not your password). Same procedure as TestPyPI.

---

## I. After PyPI publish

- [ ] Update `README.md` install section: replace "Once published:" with the
      actual `pipx install suture` command, no conditional wording
- [ ] Update `CHANGELOG.md`: replace `[0.1.0] - Unreleased` with
      `[0.1.0] - YYYY-MM-DD`
- [ ] Create GitHub tag: `git tag v0.1.0 && git push origin v0.1.0`
- [ ] Create GitHub Release with notes copied from `CHANGELOG.md`
- [ ] Share with early users
